# Bywater Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Bywater — a second Headwater server instance running on Caruana (RTX 4090) — with its own branding, CLI entry point, and transparent client-side host routing via a `host_alias` parameter.

**Architecture:** The server codebase is already host-agnostic; Bywater is a new CLI entry point + branded logo that boots `HeadwaterServer` with a different title. On the client side, a `host_alias: Literal["headwater", "bywater"]` param threads from client → transport → `get_network_context()`, which gains a new `bywater_server` field in `NetworkContext`. Host resolution is deferred from module-import time to instantiation time.

**Tech Stack:** Python 3.12, FastAPI, httpx, requests, pytest, monkeypatch (no new dependencies)

---

## Non-Goals

The following are explicitly out of scope. Do not add them:

- **No `BywaterClient` class.** The interface is `HeadwaterClient(host_alias="bywater")`. No separate client class.
- **No port change.** Bywater runs on 8080, same as Headwater. They are on different hosts. Do not use 8081.
- **No `is_bywater_server` field on `NetworkContext`.** Caruana is both the database host and the Bywater server host. `is_database_server` covers both roles.
- **No `base_url` parameter on sync `HeadwaterClient`.** It doesn't have one today. Don't add it for "parity."
- **No `conftest.py` or shared test fixtures.** The `_fake_ctx` helper is inlined in each test file. Each test file is self-contained.
- **No systemd service file.** Infrastructure setup on Caruana is out of scope.
- **No `host_alias` in `__repr__` or `__str__` of transport classes.**
- **No ChromaDB configuration.** Server-side service configuration on Caruana is out of scope.

---

## Acceptance Criteria

- **AC-1:** `NetworkContext` dataclass has a `bywater_server: str` field
- **AC-2:** `get_network_context()` populates `bywater_server` correctly for all five network scenarios: on-Caruana (localhost), on-AlphaBlue (172.16.0.4), VPN (172.16.0.4), LAN-only (10.0.0.82), remote fallback (68.47.92.102 hardcoded — same as existing `siphon_server` fallback, not `public_ip`)
- **AC-3:** Both transport classes call `get_network_context()` at instantiation time, not at module import; a post-import monkeypatch of `get_network_context` affects new transport instances
- **AC-4:** `HeadwaterTransport` accepts `host_alias: Literal["headwater", "bywater"] = "headwater"` and resolves `base_url` from `siphon_server` or `bywater_server` accordingly; explicit `base_url` overrides both; `_get_url()` emits a `DEBUG` log naming the alias and resolved IP
- **AC-5:** `HeadwaterAsyncTransport` accepts the same and behaves identically, including the `DEBUG` log
- **AC-6:** `HeadwaterClient` accepts `host_alias: Literal["headwater", "bywater"] = "headwater"` and threads it to `HeadwaterTransport`
- **AC-7:** `HeadwaterAsyncClient` accepts `host_alias` alongside existing `base_url`; explicit `base_url` still takes precedence even when `host_alias="bywater"` is also specified
- **AC-8:** `HeadwaterServer.__init__` accepts `name: str = "Headwater API Server"`, uses it as the FastAPI app title, and uses it in the startup/shutdown log lines
- **AC-9:** A `bywater` CLI entry point exists; invoking it prints the BYWATER logo and starts the server with `name="Bywater API Server"` on port 8080
- **AC-10:** `print_bywater_logo()` exists in `logo.py`, outputs a banner in green (`\033[92m`) that is byte-for-byte different from the output of `print_logo()`
- **AC-11:** `GET /status` returns a `server_name` field equal to the `name` passed to `HeadwaterServer.__init__`; default is `"Headwater API Server"`; Bywater returns `"Bywater API Server"`

---

## Known Gaps (Out of Scope Here)

- Transport instantiation now triggers up to ~17s of network I/O in worst-case (not on VPN, not on LAN, all public IP services time out). Do not call `HeadwaterTransport()` / `HeadwaterAsyncTransport()` in hot paths or request handlers. Instantiate once at startup.

```python
def get_network_context() -> NetworkContext:
    """
    Priority order:
    1. On database server / Bywater host (Caruana) -> localhost
    2. On siphon server / Headwater host (AlphaBlue) -> peer VPN IPs
    3. Connected to VPN -> use VPN IPs (172.16.0.x)
    4. On local network -> use LAN IPs (10.0.0.x)
    5. Remote -> use hardcoded public WAN IP (fallback, requires exposed ports)

    Note: Caruana serves as both the database server (is_database_server) and
    the Bywater server host. These roles share the same hostname check.
    """
    hostname = get_hostname()
    is_database_server = hostname == "Caruana"
    is_siphon_server = hostname == "AlphaBlue"
    private_ip = get_private_ip()
    public_ip = get_public_ip()
    vpn_ip = get_vpn_ip()

    vpn_connected = is_on_vpn()
    local_network = is_on_local_network()

    if is_database_server:
        preferred_host = "localhost"
        siphon_server = "172.16.0.2"
        bywater_server = "localhost"
    elif is_siphon_server:
        preferred_host = "172.16.0.4"
        siphon_server = "localhost"
        bywater_server = "172.16.0.4"
    elif vpn_connected:
        preferred_host = "172.16.0.4"
        siphon_server = "172.16.0.2"
        bywater_server = "172.16.0.4"
    elif local_network:
        preferred_host = "10.0.0.82"
        siphon_server = "10.0.0.87"
        bywater_server = "10.0.0.82"
    else:
        preferred_host = "68.47.92.102"
        siphon_server = "68.47.92.102"
        bywater_server = "68.47.92.102"  # hardcoded WAN IP, same as existing fields

    return NetworkContext(
        local_hostname=hostname,
        is_on_vpn=vpn_connected,
        is_local=local_network,
        is_database_server=is_database_server,
        is_siphon_server=is_siphon_server,
        preferred_host=preferred_host,
        siphon_server=siphon_server,
        bywater_server=bywater_server,
        public_ip=public_ip,
        local_ip=private_ip,
        vpn_ip=vpn_ip,
    )
```

- [ ] **Step 4.3 — Implement `host_alias` and debug log in `HeadwaterTransport`**

Replace the module-level declarations, `__init__`, and `_get_url` in `headwater_transport.py` with the following. Leave `_handle_error_response`, `_request`, `ping`, `get_status`, `list_routes`, `get_logs_last` unchanged.

```python
from __future__ import annotations

from typing import Literal

from headwater_api.classes import (
    HeadwaterServerError,
    HeadwaterServerException,
    LogsLastResponse,
    StatusResponse,
)
from dbclients.discovery.host import get_network_context
from urllib.parse import urljoin
import requests
import logging
import json

logger = logging.getLogger(__name__)

HEADWATER_SERVER_DEFAULT_PORT = 8080


class HeadwaterTransport:
    """
    Transport layer for communicating with HeadwaterServer.
    """

    def __init__(
        self,
        base_url: str = "",
        host_alias: Literal["headwater", "bywater"] = "headwater",
    ):
        self._host_alias = host_alias
        if base_url == "":
            self.base_url: str = self._get_url()
        else:
            self.base_url = base_url.rstrip("/")
        self._session = requests.Session()

    def _get_url(self) -> str:
        ctx = get_network_context()
        ip = ctx.bywater_server if self._host_alias == "bywater" else ctx.siphon_server
        url = f"http://{ip}:{HEADWATER_SERVER_DEFAULT_PORT}"
        logger.debug(f"[{self._host_alias}] resolved to {ip}:{HEADWATER_SERVER_DEFAULT_PORT}")
        return url
```
